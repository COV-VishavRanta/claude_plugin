# Vue 3 Development Instructions

## Core Principles

- **Composition API**: Always use `<script setup lang="ts">` for all components.
- **Language**: Use TypeScript for all components, stores, composables, and services.
- **Style Guide**: Follow the [Official Vue Style Guide](https://vuejs.org/style-guide/) priorities A and B.

## Essential Rules (Vue Style Guide Priority A)

- **Multi-word component names**: Always use multi-word names (e.g., `TodoItem`, `UserProfile`) to avoid conflicts with HTML elements. Only the root `App` component is exempt.
- **Keyed `v-for`**: Always provide a unique `:key` on `v-for` iterations.
- **No `v-if` with `v-for`**: Never use `v-if` on the same element as `v-for`. Use a computed property to filter the list, or wrap in a `<template v-for>`.

## Strongly Recommended Rules (Vue Style Guide Priority B)

- **Base component prefix**: Presentational/dumb components use a consistent prefix — `Base`, `App`, or `V` (e.g., `AppDropdown`, `BaseTable`).
- **Tightly coupled names**: Child components that only exist within a parent include the parent name as a prefix (e.g., `TodoListItem`, `SearchSidebarNavigation`).
- **Self-closing components**: Use `<MyComponent/>` when a component has no content.
- **Multi-attribute elements**: Place one attribute per line when an element has two or more attributes.
- **Simple template expressions**: Move complex logic into `computed` properties; keep templates declarative.
- **Directive shorthands**: Use `:` instead of `v-bind:`, `@` instead of `v-on:`, and `#` instead of `v-slot:` — consistently throughout the project.
- **Full-word names**: Prefer full words over abbreviations in component names (e.g., `StudentDashboardSettings` not `SdSettings`).

## Component Best Practices

- **Naming**: Use PascalCase for component filenames and references (e.g., `MyComponent.vue`).
- **Reactivity**: Use `ref()` for primitive values and `reactive()` for objects/complex structures.
- **Two-Way Binding**: Use `defineModel()` for simplified `v-model` implementation (Vue 3.4+).
- **Props/Emits**: Use `defineProps` and `defineEmits` with strict TypeScript interfaces for component contracts.
- **Composables**: Extract reusable logic into composables (e.g., `useMyFeature.ts`). Never use mixins.
- **Size Limit**: Keep components under 300 lines; refactor into smaller components if they grow larger.
- **Template refs**: Prefer template refs over direct DOM access.
- **Business logic**: Keep business logic out of templates; use computed properties, composables, or store actions.

## Component Auto-Registration

- `components.d.ts` is auto-generated by `unplugin-vue-components`.
- Shared components placed in `src/components/` are globally registered — no manual imports needed.
- Use components directly in templates (e.g., `<FormInput>`, `<AppDropdown>`); the declaration file provides type safety.
- When creating a new shared component, place it in the correct `src/components/` subfolder — `components.d.ts` updates automatically when the dev server runs.

## Step Sub-Component Organization

Top-level step components are standalone files. Step-specific child components live in a lowercase subfolder matching the step name:

```
src/views/<Feature>/steps/
├── StepOne.vue                   ← top-level step component
├── StepTwo.vue
├── stepOne/                      ← step-specific sub-components
│   ├── ConditionalSection.vue
│   └── DetailCard.vue
├── stepTwo/
│   ├── ItemCard.vue
│   └── AddItemModal.vue
```

- Step-specific sub-components go in the lowercase subfolder.
- Reusable components shared across steps go in `src/components/shared/`.
- Modals within steps have their own isolated form instances, not part of the parent form.

## Pinia State Management

- **Setup Stores**: Prefer "Setup Stores" (composition function) over "Option Stores" for alignment with the Composition API.
- **Reactivity**: Use `storeToRefs()` when destructuring state to maintain reactivity.
- **Actions**: Access store actions directly via the store instance (e.g., `store.increment()`) rather than dispatches.
- **Scope**: Use Pinia only for global state; keep local state within components using `ref` or `reactive`.
- Expose only necessary state via `storeToRefs()` in components.

## Styling

- **Scoped CSS**: Always use `<style scoped>` to prevent style leakage.
- **Preprocessor**: Use SCSS for styling by default.
- **Imports**: Use `@use` instead of `@import`.
- **Component-specific styles**: When styles go beyond `<style scoped>` (deep overrides, shared theming), create a dedicated file at `src/styles/components/component-<name>.scss` using the `component-` prefix with kebab-case naming.

## Composables & Code Patterns

- **Export Style**: Prefer arrow-function exports for composables and small default-export factories.
- **Shared Logic**: Reuse composables (e.g., `useFormatDate`, `useTranslation`) instead of duplicating code.
- **Dual exports**: Composables that are mostly pure functions may export both the utility function (e.g., `formatUSPhone`) and a composable wrapper (e.g., `usePhoneFormat`).

## Form Data Handling

- **Destructure `Record<string, unknown>` sources once.** Never access `data.fieldName` or `formData.value.fieldName` repeatedly. Destructure at the top of the function and use plain variable names throughout.
- **Use type-safe coercion helpers** (e.g., `getString(value)`, `getBoolean(value)` from `@/helpers`) to safely coerce unknown values. Never use inline `(value as string) || ''` casts.
- **Organize schemas and payloads** in centralized folders under the feature view:
  - `views/<Feature>/schemas/<stepNameSchema>.ts` — Zod validation schemas + form initialization factories
  - `views/<Feature>/payloads/<stepNamePayload>.ts` — API payload builders (initial state + transformation mapper)
- **Extract `initialValues` factories** into the schema file (e.g., `getStepOneInitialValues`). Do not inline them in `useForm()` calls inside components.
- **Extract payload builders** into the payloads file (e.g., `createStepOneInitialState` and `mapStepOnePayload`). The store calls them — it never contains raw field-mapping or payload transformation logic.
- **Pattern for adding a new step:**
  1. `views/<Feature>/schemas/<stepNameSchema>.ts` — Zod schema + `get<StepName>InitialValues(data)` factory
  2. `views/<Feature>/payloads/<stepNamePayload>.ts` — `create<StepName>InitialState()` + `map<StepName>Payload(data)`
  3. Store: import both, call `create<StepName>InitialState()` in the default payload, call `map<StepName>Payload(formData.value)` in the sync function

## API Service Conventions

- **Singleton HTTP client**: Use a single shared HTTP client instance — never create ad-hoc Axios or fetch instances.
- **Domain service objects**: Organize API calls into domain-specific service objects (e.g., `UserAPIService`, `OrderAPIService`) with typed async methods.
- **Error messages**: Centralize error messages in constants — never use inline error strings.
- **Interceptors**: Handle cross-cutting concerns (auth headers, token expiration, 401 redirects, error toasts) in request/response interceptors.
- **Response typing**: Type API responses with a generic wrapper (e.g., `ApiResponse<T>`).

## Router Conventions

- **Enum-based routes**: Use enums or constants for route names and paths — never hardcode route strings in components.
- **Lazy loading**: Lazy-load all route components: `component: () => import('@/views/...')`.
- **Route meta**: Use route `meta` for cross-cutting concerns (auth requirements, breadcrumbs, page titles).
- **Middleware**: Implement navigation guards as separate middleware functions, not inline callbacks.

## Project Architecture & File Structure

### Folder Structure

```
src/
├── api/              # API call functions (separation of concerns, reusability)
├── assets/           # Static assets
│   ├── images/       # Image files (.jpg, .svg, .webp, etc.)
│   └── styles/       # Global CSS/SCSS files
├── components/       # Reusable Vue components (MVVM View pattern)
│   ├── Layouts/      # Layout components (e.g., MainHeader, MainFooter)
│   └── UIs/          # Base/shared UI components (e.g., BaseButton, BaseInput)
├── composables/      # Composable functions (custom hooks using Composition API)
├── constants/        # Project-wide constant values
├── enums/            # TypeScript enum definitions
├── helpers/          # Global utility/helper functions
├── plugin/           # Vue plugin registrations (third-party libraries/packages)
├── router/           # Vue Router configuration and route definitions
├── stores/           # Pinia state management stores
└── views/            # Page-level view components (MVVM architecture)
    └── <Feature>/    # Feature folder (PascalCase)
        ├── <Feature>View.vue or <Feature>ViewModel.vue
        ├── Sections/          # Sub-sections composing the view
        └── components/        # Components specific to this view only
```

### Folder & Naming Conventions

- **api/**: One file per feature in camelCase (e.g., `products.ts`). Function names in camelCase.
- **assets/**: Folder names lowercase. Image filenames all lowercase, no spaces or underscores, use hyphens to separate words. CSS filenames lowercase, short, no special characters.
- **components/**: Folder names in PascalCase. `.vue` files in PascalCase. Sub-folders like `Layouts/` and `UIs/` can be added or removed per project needs.
- **composables/**: Files in `.ts` format. Name functions with `use` prefix in camelCase (e.g., `useFetch`, `useMouse`).
- **constants/**: File named by feature in PascalCase (e.g., `Common.ts`). If a constant is only used in one component, keep it in that component file.
- **enums/**: Files in lowercase `.ts`. Enum names in PascalCase, keys in UPPERCASE. If an enum is only used in one component, keep it in that component file.
- **helpers/**: File named per feature (e.g., `utility.ts`). Function names in camelCase.
- **plugin/**: File names in camelCase. Register all Vue third-party libraries/packages here.
- **router/**: Routes registered in `index.ts`. Additional route files named by feature in camelCase only when the number of routes is overwhelming.
- **stores/**: File names in camelCase. Each store needs a unique `id`. Exported store function prefixed with `use` (e.g., `useCounterStore`).
- **views/**: Feature folders in PascalCase. View files end with `View` or `ViewModel` suffix in PascalCase (e.g., `DashboardView.vue`, `HomeViewModel.vue`). Use `Sections/` sub-folder for partial views. Use a local `components/` sub-folder for view-specific components.

### General Rules

- **Helpers**: Centralize small reusable helpers (readers, formatters, validators) in `src/helpers` or `src/composables`.
- **Constants**: Centralize constants (pagination options, page size, date formats) in `src/constants`.
- **UI Configs**: Keep table field definitions and UI configs in `src/views/tableFields` instead of inline literals.
- **Icons**: Store all icon assets strictly in the designated icons folder.
- **API Layer**: Centralize API response normalization in the service layer.

## Security

- Never use `v-html` with untrusted content; sanitize with DOMPurify when HTML rendering is required.
- Never bind unsanitized user input to `href`, event handler attributes (`onclick`, `onfocus`), or `:style`.
- Never use user-provided strings as component templates.

## General Best Practices

- **Custom Directives**: Use the `v-` prefix for custom directives and implement them in the `directives` folder.
- **State Management**: Use Pinia for global state management; avoid the deprecated Vuex.
- **Accessibility**: Include necessary `aria-*` attributes and semantic HTML tags in all templates.
- **Performance**: Use `v-show` for frequent toggling and `v-if` for conditional rendering that rarely changes.
- **Avoid unnecessary abstractions**: Do not create wrapper components or higher-order components unless they are used in multiple places.
